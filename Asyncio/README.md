## Ключевые понятия (мини-словарь)

Прежде чем отвечать на вопросы, разберитесь с этими словами. Найдите определение
каждого и запишите своими словами:

**Конкурентность (concurrency)** vs **параллелизм (parallelism)** 

- **Конкурентность (concurrency) vs параллелизм (parallelism).**
Конкурентность — это *структура* программы, при которой несколько задач **продвигаются вперёд за один и тот же период времени**, по очереди уступая друг другу управление. Они могут выполняться на одном ядре: пока одна задача ждёт (например, ответа сети), управление получает другая. Параллелизм — это *физически одновременное* исполнение на нескольких ядрах/процессорах в один и тот же момент. Грубо: конкурентность — «жонглировать несколькими делами», параллелизм — «несколько рук делают дела одновременно». asyncio даёт **конкурентность без параллелизма** (один поток), `multiprocessing` — настоящий параллелизм.

**I/O-bound** **CPU-bound**:

- **I/O-bound** — в основном ждём (сеть, диск, БД). 

- **CPU-bound** — в основном считаем (матрицы, хеши, видео).

**Блокирующий** и **неблокирующий** вызов.

- **Блокирующий** вызов — не отдаёт управление (`time.sleep`). **Неблокирующий** — отдаёт (`await asyncio.sleep`).

**Event loop (цикл событий)** — кто и как его запускает в asyncio.

- Event loop (цикл событий).
Это «диспетчер» asyncio — **бесконечный цикл**, который держит очередь готовых к выполнению задач, по очереди передаёт им управление, а на await забирает управление обратно и переключается на другую готовую задачу. В современном коде его запускает asyncio.run(main()): он создаёт loop, крутит его до завершения корутины main и закрывает. Раньше делали вручную: loop = asyncio.get_event_loop(); loop.run_until_complete(...).

**Корутина (coroutine)**, **Task**, **Future** — чем отличаются.

- **Корутина** — результат `async def`, сама не бежит. **Task** — корутина, запущенная в loop (`create_task`). **Future** — «обещание результата», Task — её частный случай.

**Кооперативная многозадачность** — почему она «кооперативная»?

- «Кооперативная» — потому что задачи **сами добровольно** уступают управление в точках `await`. Никто не вытесняет их принудительно (в отличие от потоков ОС, которые планировщик прерывает в произвольный момент). Если корутина не делает `await` и захватила процессор тяжёлым расчётом — она **никому не отдаёт управление**, и весь event loop «висит». Сотрудничество держится на честности кода.

> 💡 Подсказка: ключевые слова для поиска — `async def`, `await`, `asyncio.run`,
`asyncio.gather`, `asyncio.create_task`.

### 3. Что такое GIL? Как он устроен?

Расшифруйте аббревиатуру GIL.:

- **Расшифровка.** GIL — **Global Interpreter Lock**, «глобальный замок интерпретатора». Это правило внутри Python: **в один момент Python-код выполняет только один поток**. Остальные ждут своей очереди.
 

🔎 Какую проблему GIL решает (зачем он вообще нужен интерпретатору)?

- **Зачем он нужен.** Python сам следит за памятью: у каждого объекта есть счётчик «сколько на меня ссылаются», и при нуле объект удаляется. Если бы несколько потоков меняли эти счётчики **одновременно**, они бы запутались — память текла бы или удалялись бы ещё нужные объекты (краши). GIL — простой способ это предотвратить: «к памяти подходим по одному». Плюс — упрощает жизнь библиотекам на C.


🔎 Сколько потоков Python могут одновременно исполнять байт-код при наличии GIL?

- **Один.** Поэтому несколько потоков на тяжёлых вычислениях не дают ускорения.

🔎 Когда поток **освобождает** GIL? (подсказка: операции ввода-вывода, тайм-слайсы)

- Когда уходит **ждать ввод-вывод** (сеть, диск) — «я всё равно жду, держите замок». Поэтому для **ожиданий** потоки всё-таки полезны.
- **По таймеру** — примерно каждые 5 мс Python отбирает замок и даёт шанс другим потокам.
- В **C-библиотеках** (как numpy), которые специально отпускают замок на время тяжёлой работы.

🔎 Слышали ли вы про «GIL-free» Python (PEP 703, free-threading)? Что это и зачем?

- **GIL-free Python (PEP 703).** Это сборка Python **без этого замка**, чтобы потоки наконец работали **по-настоящему параллельно** на нескольких ядрах. Где оно сейчас: в Python 3.13 появилось как эксперимент, а <cite index="5-1">в Python 3.14 такая сборка перестала быть экспериментальной, хотя по умолчанию её всё ещё не ставят</cite> — это <cite index="8-1">отдельная программа `python3.14t` (буква `t` на конце), которую надо ставить специально</cite>. Обычный Python пока по-прежнему с замком. Расплата за свободу — <cite index="6-1">обычный однопоточный код стал медленнее примерно на 5–10%</cite>.

```python
# 03_gil.py — потоки НЕ ускоряют вычисления из-за GIL
import time
from threading import Thread
 
def cpu_heavy(n):
    s = 0
    for i in range(n):
        s += i * i
    return s
 
N = 20_000_000
 
start = time.perf_counter()
cpu_heavy(N); cpu_heavy(N)
print(f"По очереди: {time.perf_counter()-start:.2f}с")
 
start = time.perf_counter()
t1 = Thread(target=cpu_heavy, args=(N,))
t2 = Thread(target=cpu_heavy, args=(N,))
t1.start(); t2.start(); t1.join(); t2.join()
print(f"В 2 потоках: {time.perf_counter()-start:.2f}с")
```
«В 2 потоках» получается **не быстрее**, чем по очереди — вот он, эффект замка. (На сборке `python3.14t` два потока ускорили бы расчёт по-настоящему.)

### 4. Что происходит при обращении к генератору функциями `send`, `next`, `throw`?

Это фундамент, на котором исторически были построены корутины.

Генератор — это функция, которая умеет **ставиться на паузу** на слове `yield` и потом продолжать с того же места. Именно из этой идеи когда-то выросли корутины.
 
- **`next(gen)`** — «продолжай до следующей паузы (`yield`) и покажи, что выдал». Первый раз — с начала функции; дальше — с места последней паузы.
- **`gen.send(value)`** — то же самое, но ещё и **передаёт значение внутрь**. Это значение «прилетает» туда, где генератор стоял на `yield`. `send(None)` = то же, что `next()`.
- **`gen.throw(...)`** — «подкинь генератору ошибку прямо туда, где он стоит». Внутри её можно поймать через `try/except`.
- **`yield` как выражение** (`x = yield`) — это «улица с двусторонним движением»: генератор отдаёт значение наружу и засыпает, а когда его будят через `send(data)`, то `data` **становится** результатом этого `yield`.
```python
# 04_generator.py — потрогать руками next / send / throw
def echo():
    print("генератор стартовал")
    try:
        while True:
            received = yield          # сюда прилетит то, что отправим через send()
            print(f"получил: {received}")
    except ValueError as e:
        print(f"поймал ошибку: {e}")
        yield "завершаюсь"
 
gen = echo()
print("1)", next(gen))                # дошёл до yield, выдал None
print("2)", gen.send(42))             # 42 прилетело в received
print("3)", gen.send("hi"))           # "hi" -> received
print("4)", gen.throw(ValueError("стоп")))  # ошибку поймал except
```
Вывод (проверено):
```
генератор стартовал
1) None
получил: 42
2) None
получил: hi
3) None
поймал ошибку: стоп
4) завершаюсь
```
Что произошло: (1) `next()` довёл до первой паузы. (2) `send(42)` положил 42 в `received`, напечатал и снова дошёл до паузы. (3) то же с `"hi"`. (4) `throw` подкинул ошибку прямо в точку паузы, её поймал `except`, и генератор напоследок выдал `"завершаюсь"`.

### 5. GIL участвует при выполнении асинхронного python кода?

🔎 asyncio использует потоки или один поток?

- **один поток** → GIL фактически не мешает (драться за него некому).

🔎 Раз весь код выполняется в одном потоке — мешает ли GIL внутри `asyncio`?

- Почти **нет**. Замок мешает, когда **несколько потоков** дерутся за право выполнять код. А здесь поток один — драться не за что. В каждый момент и так работает одна задача.

🔎 Если внутри корутины запустить тяжёлый **CPU-bound** расчёт без `await` что станет с остальными корутинами? Причём тут GIL, а причём — event loop?

- **Тяжёлый CPU-bound без `await` → весь loop замирает.** Виноват **event loop** (переключение только на `await`), а не GIL.


🔎 Как «подружить» asyncio с CPU-bound кодом? (подсказка:
`run_in_executor`, `ProcessPoolExecutor`)

- `loop.run_in_executor(ProcessPoolExecutor(), func, *args)` — расчёт уходит в **отдельный процесс** (свой GIL → настоящий параллелизм, обходит ограничение GIL). Правильный выбор для CPU-bound.
- `ThreadPoolExecutor` — годится, чтобы не блокировать loop **блокирующими I/O-вызовами** (где GIL отпускается), но для чистого CPU из-за GIL ускорения не даст.
- Удобная обёртка: `await asyncio.to_thread(func, *args)` (поток) — отлично для блокирующего I/O.
```python
# 05b_executor.py — CPU-bound в отдельном процессе, loop остаётся живым
import asyncio, time
from concurrent.futures import ProcessPoolExecutor
 
def cpu_heavy(n):
    return sum(i * i for i in range(n))
 
async def heartbeat():
    for _ in range(5):
        print("loop живой...")
        await asyncio.sleep(0.3)
 
async def main():
    loop = asyncio.get_running_loop()
    start = time.perf_counter()
    with ProcessPoolExecutor() as pool:
        calc = loop.run_in_executor(pool, cpu_heavy, 30_000_000)  # уходит в процесс
        result, _ = await asyncio.gather(calc, heartbeat())       # loop не блокируется
    print(f"результат={result}, время={time.perf_counter()-start:.2f}с")
 
asyncio.run(main())
```
Теперь heartbeat'ы идут **во время** расчёта: тяжёлая работа исполняется в отдельном процессе, а event loop в основном процессе свободно обслуживает другие корутины.
